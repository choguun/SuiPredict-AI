R57 SDK Audit Findings — 2026-06-06
====================================

Executive summary
-----------------
Audited packages/sdk/src/ (post-R56 sweep) for end-to-end integration
gaps. The R56 fixes (env `.trim()`, runtime outcome/tier/legIndex
validation, lowercase normalization, BigInt handling, dotAll regex,
quoteType type shape) closed the obvious classes of bug. R57 focuses
on the survivors: the *consistency* of the fix pattern, the *coverage*
of the parlay/vault type-arg validation, and a few latent crash /
drift / mis-shape hazards that the R56 sweep didn't touch.

Found 18 high-quality issues:

  CRITICAL: 1
  HIGH    : 5
  MEDIUM  : 7
  LOW     : 5

Top survivors:
- Five of the six SDK files still carry the *tautological* `PKG()`
  getter (`const PKG = () => AGENT_POLICY_PACKAGE_ID`) that R56.12
  flagged on the sixth (`prediction-market-client.ts`). The R55
  `resolveAgentPolicyPackageId()` getter was added to make the
  function-form hot-patchable, but the 5 sibling files only updated
  the *shape* (function-arrow), not the *body* (still return the
  frozen module-level const). Operators who hot-patch the env
  via `bootstrap-env.ts` see zero effect on the parlay/streak/
  prize/badge/profile builders.
- Seven parlay builders (`buildCreateParlayPoolTx`,
  `buildFundParlayPoolTx`, `buildParlayAdminWithdrawTx`,
  `buildSetMaxPayoutBpsTx`, `buildCreateParlayTx`,
  `buildRecordLegTx`, `buildFinalizeParlayTx`) take a `coinType:
  string` for the generic `Q` but never validate it. The R54
  `buildClaimPrizeTx` validates `prizeCoinType.startsWith("0x")`;
  the seven parlay siblings silently forward `""` or a
  space-padded type tag to `typeArguments` and let the wallet
  surface "Invalid type argument". The on-chain `parlay::create_pool<Q>`
  is generic over `Q`; an empty type arg would be a BCS encoder
  type-argument-count mismatch.
- The `extractMoveAbortCode` regex (`/MoveAbort[\s\S]*?\)\s*,\s*(\d+)\s*\)/`)
  is non-greedy, which means it returns the abort code of the
  **innermost** `MoveAbort(...)` in a nested error message. Sui's
  gRPC `FailedTransaction` rendering nests aborts (a
  `prediction_market::redeem` that internally aborts on
  `balance_manager::withdraw` produces two `MoveAbort` blocks), and
  the agents' `isMoveAbortCode(..., "prediction_market", 9)` check
  then always returns false on the outer call, silently misrouting
  the failure to the "unknown abort" branch.

Findings
--------

[CRITICAL] R57.1 | 5 SDK files | `const PKG = () => AGENT_POLICY_PACKAGE_ID` is a tautology — R56.12 only fixed one of six
  Where: packages/sdk/src/parlay-client.ts:29
         packages/sdk/src/badge-nft-client.ts:23
         packages/sdk/src/streak-client.ts:22
         packages/sdk/src/prize-client.ts:20
         packages/sdk/src/user-profile-client.ts:26
  What: R55 added `resolveAgentPolicyPackageId()` in
        `constants.ts:331-339` for hot-patch sensitivity. R56.12
        then audited the local `PKG()` getters across the SDK
        and fixed the one in `prediction-market-client.ts:67` to
        read `const PKG = () => resolveAgentPolicyPackageId()`. The
        five sibling files were updated only to the *function* form
        (e.g. `const PKG = () => AGENT_POLICY_PACKAGE_ID;`) — the
        body still returns the frozen module-level const, not the
        hot-patchable getter. An operator who runs
        `bootstrap-env.ts` to swap the deployed `prediction_market`
        package (e.g. for an emergency redployment after a Move
        bug) sees the new address picked up by the prediction-
        market builders but **not** by the parlay, badge, streak,
        prize, or user-profile builders — every PTB built by those
        five files then routes to the old (now-non-existent) package
        and aborts with `package object not found`.
  Why:  CRITICAL because the function-arrow shape is *explicitly*
        the hot-patch signal the operator relies on. R56.12's
        docstring on the fix says:
          "The getter-as-function shape was already correct; the
           body was the bug."
        That statement is true for `prediction-market-client.ts`
        and false for the other five — the bodies there are still
        the original tautology. The agents' `market-creator`,
        `parlay-worker`, `prize-distributor`, and `prize-admin`
        all import these builders and the new package is invisible
        to all of them post-patch.
  How:  Mirror the R56.12 fix in each of the five files. In
        `parlay-client.ts`, `badge-nft-client.ts`,
        `streak-client.ts`, `prize-client.ts`, and
        `user-profile-client.ts`:
          - import `resolveAgentPolicyPackageId` from `./constants.js`
            (replacing the bare `AGENT_POLICY_PACKAGE_ID` import
            where applicable),
          - change `const PKG = () => AGENT_POLICY_PACKAGE_ID;` to
            `const PKG = () => resolveAgentPolicyPackageId();`.
        A follow-up should add a shared `PKG()` helper in
        `utils.ts` (one getter, one place) so a future R-sweep
        can't leave the same class of bug behind in a new sibling
        file.

[HIGH] R57.2 | 7 parlay builders | `coinType` arg is forwarded to `typeArguments` without validation
  Where: packages/sdk/src/parlay-client.ts:59  (buildCreateParlayPoolTx)
         packages/sdk/src/parlay-client.ts:94  (buildFundParlayPoolTx)
         packages/sdk/src/parlay-client.ts:145 (buildParlayAdminWithdrawTx)
         packages/sdk/src/parlay-client.ts:210 (buildSetMaxPayoutBpsTx)
         packages/sdk/src/parlay-client.ts:261 (buildCreateParlayTx)
         packages/sdk/src/parlay-client.ts:382 (buildRecordLegTx)
         packages/sdk/src/parlay-client.ts:451 (buildFinalizeParlayTx)
  What: Each builder takes a `coinType: string` arg and passes it
        verbatim to `typeArguments: [coinType]`. None of the seven
        validates the string. R54 added a `prizeCoinType` check
        to `buildClaimPrizeTx` (`prize-client.ts:282-287`):
          if (!prizeCoinType || !prizeCoinType.startsWith("0x")) {
            throw new Error("...");
          }
        The parlay siblings are inconsistent with this — a
        caller who passes `coinType: ""` (the default for
        `String(undefined)`) produces a `typeArguments: [""]`
        PTB, which the Sui BCS encoder rejects with a
        type-argument-count mismatch, surfaced inside the wallet
        as a generic "Invalid type argument". The on-chain
        `parlay::create_pool<Q>`, `parlay::fund_pool<Q>`, etc.
        are generic over `Q`; an empty `Q` is not a valid
        coin type.
  Why:  The parlay admin path is script-driven (the agents
        `bootstrap-parlay.ts` and `prize-admin.ts` call these
        with values read from env). A misconfigured
        `PARLAY_COIN_TYPE` env var (or a script that forgot to
        read the env) silently ships a malformed PTB. The
        prize-client sibling has the guard; the parlay-client
        siblings don't. Inconsistency is the bug here, not
        the user error.
  How:  Add a shared `validateCoinType(coinType)` helper in
        `utils.ts` (mirroring the existing R54 logic in
        `prize-client.ts:282-287`) and call it at the top of
        each of the seven builders. Reject `!coinType`,
        missing `0x` prefix, missing `::` separator, or
        trailing whitespace (the last catches the R41-style
        env-paste bug). The `prize-client.ts:282-287` check
        should also be migrated to the same helper for
        consistency.

[HIGH] R57.3 | packages/sdk/src/move-errors.ts:173 | `extractMoveAbortCode` regex returns the *innermost* abort code, not the outermost
  Where: packages/sdk/src/move-errors.ts:159-175
  What: The regex is
          /MoveAbort[\s\S]*?\)\s*,\s*(\d+)\s*\)/
        — the `[\s\S]*?` is non-greedy, so the engine returns the
        *first* closing `, N)` it finds. Sui's gRPC
        `FailedTransaction.status.error.message` rendering
        nests `MoveAbort` blocks when a wrapper aborts on an
        inner function abort (e.g. a `prediction_market::redeem`
        that internally aborts on `balance_manager::withdraw`
        produces
          "MoveAbort(MoveLocation { module: \"balance_manager\" }, 7)
           in command 1
           MoveAbort(MoveLocation { module: \"prediction_market\" }, 9)
           in command 0"
        ). The current regex returns `7` (the inner
        `balance_manager` code), not `9` (the outer
        `prediction_market` code the caller asked about). The
        agents' `isMoveAbortCode(err, "prediction_market", 9)`
        (used in `parlay-worker.ts` and `market-resolver.ts`
        to decide retryable vs permanent failure) then returns
        `false` on the *outer* EInvalidQuantity abort and
        misroutes the PTB to the retry-with-backoff path,
        burning gas and CPU on a permanent failure.
        R56.15 fixed the multi-line `MoveLocation` rendering
        by replacing `[^\n]*` with `[\s\S]*?`; that fix is
        correct for *one* abort block but introduces the
        *innermost-wins* bug for *nested* abort blocks.
  Why:  The error translator is the single source of truth for
        "is this a permanent move-abort" across the web
        (dispute page, redeem page) and the agents (parlay-
        worker, prize-distributor, market-resolver). A wrong
        code returns means wrong retry classification, wrong
        operator alert, wrong on-call page-out.
  How:  Anchor the regex to the *last* `, N)` in the message.
        Move's renderer always puts the `MoveAbort(...)` block
        at the top of the error string, with the `command N`
        suffix, so a greedy match on `\)\s*,\s*(\d+)\s*\)` at
        the *end* of the input is the robust pattern. Replace
        the non-greedy `[\s\S]*?` with a greedy `[\s\S]*` and
        add a `(... )$` anchor:
          /MoveAbort[\s\S]*\)\s*,\s*(\d+)\s*\)\s*(?=$|in command)/
        Or, even more robustly, walk the message left-to-right
        tracking paren depth and return the code from the
        top-level (depth 0) abort. The regex form above is
        sufficient for the current SDK rendering; the walker is
        future-proof.

[HIGH] R57.4 | packages/sdk/src/predict-client.ts:716-758 | `extractCreatedObjectId` does substring match on `structSuffix` — caller-format sensitive
  Where: packages/sdk/src/predict-client.ts:716-758
  What: The function does
          types[change.objectId]?.includes(structSuffix)
        with `structSuffix` documented as e.g.
        `"::prediction_market::PredictionMarket"`. The actual
        on-chain `TypeTag` is `0x…::prediction_market::PredictionMarket`
        (full package-prefixed string). A caller who passes
        `"<PACKAGE>::prediction_market::PredictionMarket"` (the
        Move `TypeTag` syntax with a leading `<`, common in
        off-chain Move tooling) silently never matches because
        of the substring check, returns `null`, and the agents
        `market-creator.ts:240-272` re-polls with a 60s
        timeout waiting for an object that was actually
        created. The R54 audit fix correctly differentiated
        `FailedTransaction` from "object not in effects", but
        the substring match on a caller-formatted `structSuffix`
        is the surviving silent-failure path.
  Why:  The market-creator is the most-touched agent; the
        timeout-driven retry is 60s of agent stall per
        misformatted suffix. The on-chain object *is* there;
        the SDK just can't find it.
  How:  Normalize both sides of the comparison. Strip a
        leading `<`, strip a trailing `>` (the Move TypeTag
        bracket syntax), then take the substring after the
        last `::` to reduce to the bare struct name:
          const bare = structSuffix
            .replace(/^</, "")
            .replace(/>$/, "")
            .split("::")
            .pop()!;
          if (!types[change.objectId]?.includes(bare)) continue;
        This handles both the `<PKG>::module::Struct` form and
        the bare `module::Struct` form. Alternative: have the
        caller pass the bare struct name explicitly and skip
        the substring match — that requires editing the four
        agents call sites.

[HIGH] R57.5 | packages/sdk/src/markets/indexer-client.ts:104-129 | `getMarket` and `getMarketOrderBook` interpolate the `id` arg into a URL without 32-byte hex validation
  Where: packages/sdk/src/markets/indexer-client.ts:104-117 (getMarket)
         packages/sdk/src/markets/indexer-client.ts:119-130 (getMarketOrderBook)
  What: The R55 fix validates `id` is a non-empty string, then
        passes it directly to `/markets/${id}` and
        `/markets/${id}/book`. The Sui object id is `0x` + 64
        hex chars; rejecting anything else is a one-line check
        (mirror `normalizeObjectId`'s regex). A caller passing
        an `id` with embedded `/` (e.g. from a stale
        URL-param or a fuzzer) produces
        `http://localhost:3001/markets/../../admin/secrets`,
        which `fetch` will follow via the URL parser's path
        normalization. The web's `app/markets/[id]/page.tsx:40`
        reads the URL param directly; a Next.js catch-all that
        resolves to `undefined` is already caught by the R55
        check, but a *valid-looking* string with `/` (e.g.
        from a user-supplied `searchParams.q`) is not.
  Why:  The indexer is internal but the web exposes the input.
        A 5-line guard turns a path-traversal surface into a
        clean build-time error.
  How:  Add at the top of both `getMarket` and
        `getMarketOrderBook`:
          if (!/^0x[0-9a-fA-F]{64}$/.test(id)) {
            throw new Error(
              `getMarket: id must be a 32-byte hex Sui object id (got ${JSON.stringify(id)})`,
            );
          }
        Lowercase normalization is a courtesy; the indexer
        handler should accept either case.

[HIGH] R57.6 | packages/sdk/src/prize-client.ts:227-233 | `buildClaimPrizeTx` validates `rank` against `DEFAULT_DISTRIBUTION_BPS`, not the pool's actual distribution
