# R-WC-4 — v3 design spec (the proper per-market OTW design)

**Date:** 2026-06-18
**Status:** Spec only — not implemented. Live deployment is still on the v1 contract at `0xb1777f167c…`.

---

## Why v3 is needed

The v2 design (commits `408a846`, `aac7dbb`, etc.) added `phantom M: address` to `YES<Q>`, `NO<Q>`, `PredictionMarket<Q>`. The intent was that each market gets a unique coin type `YES<DUSDC, M_addr>` bypassing the Sui CoinRegistry's one-`Currency<T>`-per-package limit.

This was discovered to be **fundamentally impossible on Sui Move**:

1. **Sui's `TypeTag::Address` variant has no body** — it's a unit variant in BCS. You cannot pass `TypeTag::Address(0xMARKET)` (a concrete address instance) as a type argument. The phantom `address` only allows the primitive `address` type.
2. **Move cannot generate new types at runtime** — the `M` in `YES<Q, M>` must be a type declared at module level. There's no way to mint a unique per-market type.

The user (in chat) suggested: "M is a one-time witness type generated by the market creator, not a phantom address." This was a better suggestion but also runs into constraints — OTWs in Sui Move come from `init()` which runs once per package, not per market.

## The proper v3 design (not yet implemented)

### Per-market uniqueness without per-market types

```move
module pkg::prediction_market {
    /// Per-package one-time witness. Declared in the module, minted
    /// by `init()`. Used to register ONE `Currency<YES<Q>>` per package
    /// via `coin_registry::new_currency_with_otw`.
    public struct PackageOTW has drop {}
    
    /// YES / NO / PredictionMarket types are GENERIC on the quote
    /// coin only — there's NO `phantom M`. All markets in one package
    /// share the same `YES<Q>` coin type.
    public struct YES<phantom Q> has key, store { id: UID }
    public struct NO<phantom Q> has key, store { id: UID }
    public struct PredictionMarket<phantom Q> has key { /* ... */ }
    
    /// Per-market identity comes from a `market_id` field on the
    /// PredictionMarket struct. No phantom types.
}
```

### The bootstrap path (one-time per package)

```move
fun init(otw: PackageOTW, ctx: &mut TxContext) {
    // Register ONE YES<Q> currency type per package using the
    // module's own OTW. Subsequent create_market calls reuse this
    // currency; only the TreasuryCap is transferred per market.
    let (yes_init, yes_cap) = coin_registry::new_currency_with_otw<YES<Q>>(otw, ...);
    // ... share_object the Currency<YES<Q>>
    // ... share_object a singleton TreasuryCap<YES<Q>>
}
```

### Per-market creation

```move
public fun create_market<Q>(
    yes_cap: &mut TreasuryCap<YES<Q>>,  // shared singleton
    ...
    ctx: &mut TxContext,
) {
    // Each market gets its own PredictionMarket<Q> with a unique
    // market_id (the new UID). The YES<Q> coin type is shared.
    let market = PredictionMarket<Q> { id: object::new(ctx), ... };
    // ... register_market on the global MarketRegistry
}
```

### Why this works

- The CoinRegistry limit (one `Currency<T>` per type per package) is hit ONCE at `init()`. All subsequent markets in that package share `Currency<YES<Q>>`.
- Each market has its own `PredictionMarket<Q>` with a unique on-chain UID — per-market identity at the value level, not the type level.
- The shared `TreasuryCap<YES<Q>>` is used to mint/burn YES tokens per market, with the market's own collateral balance tracking which YES tokens belong to which market (the PTB spec is unchanged from v1).

### Trade-off

This design limits each package to one YES coin type per quote coin. Multiple packages can be deployed to support multiple quote coins. For the hackathon demo with DUSDC only, this means **one WC26 market can be on-chain per deployed package**. If the demo needs more than one, the wc-creator can publish a fresh `wc26-N` sub-package (each with its own init/OTW).

### Migration from current state

1. Revert `packages/contracts/sources/prediction_market.move` to v1 (drop `phantom M` from struct defs and function generics).
2. Drop the `parlay<M>` from `parlay.move` — keep `parlay<Q>` only.
3. Drop the SDK's `m` parameter from all builders (pass-through to v1 behavior).
4. Drop the SDK's `withMarketType` and `marketTypeSeed` helpers — no longer needed.
5. Re-publish the v3 contract as a fresh package (gets new id).
6. Update Railway + Vercel env vars to the new id.
7. Force clean rebuild of agents + web.

The wc-creator's CoinRegistry circuit-breaker would then trip after the first market per package. The wc-creator should publish N sub-packages (`wc26-A`, `wc26-B`, ...) and route each match to a different sub-package based on the market index. This is a larger refactor that was scoped out of this session.

## Honest assessment

This session has been a long exploration of multi-package testnet dynamics. The intended outcome was to publish v2 + get the wc-creator producing on-chain WC markets. That didn't happen because the v2 design was incompatible with Sui's TypeTag encoding. The correct fix is the v3 spec above, which requires either:

- (a) accepting the 1-market-per-package limit and shipping the demo with SQLite-seeded markets (current state)
- (b) deploying N sub-packages for N WC markets, which is more work but doable
- (c) a future Sui Move feature that allows concrete address type arguments (not currently supported)

I recommend (a) for the immediate demo and (b) for the post-hackathon production path.
