# HIGH — leaderboard-worker.ts:208  `countryRollup` still mutates `r.rank` in place
- File: apps/agents/src/agents/leaderboard-worker.ts:208
- Bug: After the R56 fix at line 76 changed the "happy path"
  `aggregateWeek` to build a new object via `.map((r,i)=>({...r,rank:i+1}))`
  so the rank field isn't mutated on the same instances that flow
  into `enrichRows` (which also mutates `r.claimed` /
  `r.country_code`), the `countryRollup` function still does
  `filtered.forEach((r,i)=>(r.rank = i+1))` at line 208.
- Why it matters: `countryRollup` calls `liveRollup(weekIndex, category)`
  at line 204, which already returns a freshly-mapped array (line
  181 has the same `{...r, rank: i+1}` rebuild for the category
  filter case). The re-mutation at line 208 is redundant for the
  non-filter case, and for the filtered case it mutates the same
  objects that line 181 just built. The R56 fix comment ("a future
  optimization that memoized the result across requests would be
  silently corrupted by the next leaderboard poll") applies
  equally here — `countryRollup`'s result is the second of the two
  hot REST paths the leaderboard route serves, and is hit on every
  flag click in the web UI.
- How to fix: replace the in-place mutation with
  `return filtered.map((r, i) => ({ ...r, rank: i + 1 }));` so the
  R56 invariant ("every consumer of `weekly` gets a freshly-mapped
  array, never the same `WeeklyRow` instances across requests")
  holds in the country filter path too.


# HIGH — markets/store.ts:534-543  `markMarketResolved` has no status guard
- File: apps/agents/src/markets/store.ts:534-543
- Bug: `markMarketResolved(marketId, outcome)` runs an unconditional
  `UPDATE markets SET status='resolved', outcome=? WHERE id=?`.
  The position-indexer (position-indexer.ts:509) calls it from
  `MarketResolvedEvent`; a re-poll of the same event (cursor
  regression, RPC retry, or a future redeploy that resets the
  indexer cursor without re-tagging) would silently re-stamp
  the outcome to whatever the latest event said, even if a
  `MarketUndisputedEvent` already overrode it to the dispute
  outcome (handled by `markMarketUndisputed` at line 564).
- Why it matters: A late or out-of-order `MarketResolvedEvent`
  for a market whose dispute resolved to the OTHER outcome
  would clobber `markets.outcome` and `markets.status`, breaking
  the redemption hot path (RedeemedEvent handler at
  position-indexer.ts:481 reads `market.outcome` to know which
  side to decrement — a stale `yes` for a market that the dispute
  panel flipped to `no` would attribute a winning redeem to the
  losing side). The position-indexer's own R36 + R48 fixes
  guarantee the cursor doesn't advance on a failed apply, but
  they don't guard against an out-of-order event from the chain
  reorg path.
- How to fix: add `AND status != 'resolved'` (or, more conservatively,
  `AND status NOT IN ('resolved', 'disputed')`) to the WHERE clause,
  and emit a `console.warn` if the row count is 0 so a duplicate
  event is observable. Same fix should be applied to
  `markMarketUndisputed` (line 564) to refuse a re-override of an
  already-undisputed market.

# HIGH — scheduler.ts:213-235  `msUntilNext` always walks 8*24*60 = 11520 minutes
- File: apps/agents/src/scheduler.ts:213-235
- Bug: The for-loop at line 224 iterates `i < 8 * 24 * 60 = 11520`
  minutes for ANY cron expression, including `* * * * *`
  (every minute, max 1 iteration) or `0 0 * * *` (daily at 00:00,
  max 1440 iterations). For an expression like `*/5 * * * *` (every
  5 min), the worst case is 5 iterations but the loop bound is
  still 11520.
- Why it matters: On the scheduler startup path the loop runs
  per-entry per-tick (the reschedule chain at line 188-206
  invokes `msUntilNext` each time). The waste is small for
  11 entries (max ~127k date constructions per tick), but the
  loop also runs the `candidate.getUTCMinutes()`,
  `getUTCHours()`, `getUTCDay()`, `getUTCSeconds()` parses on
  each iteration. More importantly, the function is
  intentionally simple ("walk forward to next match") which
  hides the obvious optimization: compute the next match
  arithmetically. A typo'd cron (e.g. `0 0 32 * *` day-of-month
  32) will fall through the loop and return the 8-day
  fallback, masking the typo — the boot validator at
  index.ts:256 catches the field-count typo (4 vs 5 fields)
  but not a syntactically-valid but impossible expression.
- How to fix: implement a small `nextFire(expr, now)` that
  walks only the date fields that matter (minute for `* * * * *`,
  hour for daily, dow for weekly), or terminate the loop
  early when the first `*`-wildcard field is hit. Also add
  a `console.warn` if the loop runs > 1440 iterations (means
  the expression has no match in the next 24h, which the
  R56-boot-validator doesn't catch).

# MEDIUM — position-indexer.ts:693-698  `String.fromCharCode(...number[])` for `evidence_uri`
- File: apps/agents/src/agents/position-indexer.ts:693-698
- Bug: The `MarketDisputedEvent` handler reads
  `j.evidence_uri` (declared as `string | number[]` at
  line 144) and decodes it via:
    `typeof j.evidence_uri === "string"
       ? j.evidence_uri
       : Array.isArray(j.evidence_uri)
         ? String.fromCharCode(...(j.evidence_uri as number[]))
         : ""`
  Sui renders `vector<u8>` fields as either a base64 string
  (older JSON-RPC) or a number[] (gRPC and newer JSON-RPC).
  `String.fromCharCode` interprets each number as a UTF-16
  code unit, so a base64-encoded ASCII string would
  render correctly only if every byte was <128 (true for
  ASCII, false for any UTF-8 multi-byte sequence). The
  companion helper `decodeCountryCode` at line 232
  correctly uses `Buffer.from(raw, "base64")` for the
  same problem on the `UserProfile` event.
- Why it matters: A dispute with a non-ASCII evidence URI
  (e.g. a Chinese-language URL with a `%E4%B8%AD` escape,
  or a UTF-8 path with multibyte chars) would produce
  garbled "evidence" stored in the `dispute_evidence_uri`
  column, and the UI's "View dispute evidence" link would
  404. The decoding is one-way (we don't re-encode for
  display), so a UI render in a different encoding would
  also be broken. Lower-severity than HIGH because the
  garbled string is still a unique identifier, and most
  dispute URIs in practice are ASCII.
- How to fix: use the same `Buffer.from` decode path as
  `decodeCountryCode`: if the array is `number[]`, build
  the buffer first, then `buf.toString("utf8")`. If the
  field comes as a string, treat it as base64 (not raw
  bytes) and decode via `Buffer.from(raw, "base64")`.


# MEDIUM — markets/store.ts:998  `outcome` cast to `MarketInfo["outcome"]` is unsafe
- File: apps/agents/src/markets/store.ts:997
- Bug: `outcome: (r.outcome as MarketInfo["outcome"]) ?? null`.
  The `markets.outcome` column is TEXT and is written by
  `markMarketResolved` as the literal string "yes" or "no"
  (line 542), and by `markMarketUndisputed` the same way
  (line 579). The `rowToMarket` cast doesn't validate the
  shape — a manual SQL write, a future migration that
  stashes a different value (e.g. an error string from a
  failed resolver), or a corrupted `dispute_evidence_uri`
  cross-write would surface as a non-`"yes"|"no"` value
  in the REST response. The RedeemedEvent handler at
  position-indexer.ts:481 then does
  `market.outcome === "yes" ? "yes" : "no"` — a `null`
  outcome (before resolution) goes to "no", silently
  decrementing the wrong side if the indexer's
  MarketResolvedEvent poll lagged the redeem.
- Why it matters: The pre-R36 fix shape of this code
  was the same and a different bug; the cast is a
  silent data-corruption swallower. A type-narrowing
  helper (`function asOutcome(v: unknown): "yes"|"no"|null
  { return v === "yes" || v === "no" ? v : null; }`)
  would catch the bug at the boundary and let
  RedeemedEvent bail out earlier.
- How to fix: add the helper, route the cast through it,
  log + return null on a non-`yes`/`no` value. The
  RedeemedEvent handler at position-indexer.ts:480
  already short-circuits on `!market.outcome`, so the
  indexer will skip a malformed row instead of
  decrementing the wrong side.


# MEDIUM — position-indexer.ts:286-292  cursor write is not transactional with apply
- File: apps/agents/src/agents/agents/position-indexer.ts:286-292
- Bug: `writeCursor()` does a single `INSERT OR UPDATE` on
  `indexer_state`. The R48 fix at line 357-365 deliberately
  parks the cursor when any event in the batch failed
  (returns `page.data.length` without calling
  `writeCursor`). However, the `cursor` argument to
  `writeCursor` at line 364 is `page.nextCursor` — the
  NEXT page's cursor, not the cursor for the last
  successfully applied event. If a batch has 200 events
  and 1 fails, the R48 fix correctly parks at the
  current `cursor` (the value `readCursor` returned at
  line 301) by skipping the `writeCursor` call. But
  the success path advances by the WHOLE batch, not by
  "to the last successful event". This is correct
  for the idempotent-upsert invariant, but a future
  change that adds a non-idempotent apply (e.g. a
  vault flow that adds to a balance) would silently
  re-apply events on a retry.
- Why it matters: Documented invariant of the R48 fix is
  "all current apply callbacks are idempotent via ON
  CONFLICT DO UPDATE". A future contributor adding a
  new event type with a non-idempotent apply would
  see the R48 logic and assume it's safe to retry a
  partial batch, then silently double-count. The R48
  comment at line 321-328 is explicit about this
  trade-off, but a future reader under deadline
  pressure might miss the warning.
- How to fix: advance the cursor to the last
  successful event's `event.id` (Sui cursors can
  address individual events, not just page boundaries).
  Wrap the page's apply loop in a `transaction`, and
  inside the transaction write the cursor to the
  last-applied event's id. The `page.data[i].id` is
  already populated by the SDK; the `pollAndApply`
  function just doesn't use it.

# MEDIUM — lib.ts:234-239  `closeSharedClient` returns a Promise that doesn't await on the SDK helper
- File: apps/agents/src/lib.ts:237-239
- Bug: `closeSharedClient` is exported at line 237 and
  wraps `closeClient(sdkGetSharedClient())` (line 238).
  The `closeClient` SDK helper is the same one the
  SIGTERM handler in `index.ts:636` calls via
  `closeSharedClient()`. But the wrapper signature
  is `Promise<void>` and the SDK helper's signature
  is `Promise<void>`, so this is correct on its face.
  The bug is in `getSharedClient` at line 234: the
  local wrapper returns `sdkGetSharedClient()` directly,
  but `sdkGetSharedClient` may THROW on a stale handle
  (the SDK's singleton cache is documented to throw
  if `closeClient` was already called). The local
  wrapper doesn't catch that throw, so any caller of
  `getSharedClient` after a `closeSharedClient` call
  would get an unhandled rejection.
- Why it matters: The SIGTERM handler in `index.ts:636`
  calls `closeSharedClient()` synchronously inside a
  `.then()` chain. If the process restarts the workers
  (a hot-deploy path) before `process.exit(0)` runs,
  the next worker's `getSharedClient()` would throw —
  the catch in the SIGTERM handler swallows it (line
  642 `.catch(() => {})`), so the exit succeeds, but
  a future hot-patch that re-uses the client (e.g.
  a `/run` route that runs all workers on demand)
  would crash. Low-severity because the SIGTERM path
  is the only consumer today.
- How to fix: add a try/catch in `getSharedClient` that
  resets the SDK's cache on a "closed" error and
  re-constructs. Or document the "one-shot" semantics
  with a JSDoc warning. Better: delete `closeSharedClient`
  entirely and rely on the SIGTERM handler's
  `process.exit(0)` to clean up the OS resources;
  the gRPC client's HTTP/2 keepalive pool self-cleans
  on process exit (Node's `http2` module destroys
  outstanding streams on exit).


# MEDIUM — markets/store.ts:641  `Number(row.issued)` after `RETURNING next_id - 2`
- File: apps/agents/src/markets/store.ts:638-647
- Bug: The `nextDemoOrderId` UPSERT at line 638 returns
  `RETURNING next_id - 2 AS issued`, and the consumer
  at line 647 wraps it in `Number(row.issued)`. The
  `demo_order_counters.next_id` column is INTEGER in
  the schema (per the R56 comment at line 614), so
  SQLite's better-sqlite3 driver returns it as a JS
  `number` for values < 2^53 and as a `BigInt` for
  values > 2^53 (driver default in v11+). After the
  `next_id - 2` arithmetic the value is still in the
  same domain; the `Number()` wrapper coerces
  BigInt→Number, which silently truncates above 2^53.
  A demo market that survives > 285 years of
  maker-bot ticks (`Date.now() * 1.0 + n` where
  n > 2^53) would hit the boundary, but in practice
  this is unreachable.
- Why it matters: The `Number()` wrapper documents the
  intent (the demo order_id is a JS number), but
  hides a potential silent truncation. A `BigInt`
  read of a SQLite INTEGER column is enabled by
  default in better-sqlite3 v11+; the wrapper
  undoes that safety. The R56 fix added the
  `RETURNING next_id - 2` trick specifically to
  avoid re-reading the column after the UPSERT,
  so a future refactor that changes the wrapper
  to `BigInt(row.issued)` would break the demo
  path's `order_id: number` contract in
  `market-maker.ts:218`.
- How to fix: assert the column is INTEGER and use
  `Number(row.issued)` only when the value is
  within `MAX_SAFE_INTEGER`; otherwise throw. A
  guard like `if (row.issued > BigInt(Number.MAX_SAFE_INTEGER))
  throw new Error("demo order id overflow")` would
  surface the impossible case. Or pin the column
  type to TEXT and use a string id everywhere in
  the demo path (mirroring `chain_orders.order_id`
  which is already TEXT for the same reason).

  '/.../apps/agents/dist/index.js'`. The `tsconfig.json`
  is workspace-managed and doesn't have an `outDir`
  override (defaults to `dist/`).
- Why it matters: Railway's deploy step runs `pnpm
  install && pnpm --filter @suipredict/agents build &&
  pnpm --filter @suipredict/agents start`, so the
  build is always done first. But the smoke-test
  script (`scripts/smoke-test.ts`) and the
  `verify-config.ts` script import the SDK directly,
  not the built output, so they don't exercise this
  path. A new contributor running `pnpm start` for
  the first time gets a confusing "module not found"
  error.
- How to fix: add `"start": "node dist/index.js"` →
  `"start": "node --enable-source-maps dist/index.js"`
  (so the production stack traces map back to the
  TS source) AND a `"postbuild": "echo 'built
  $(date)'"` so the build step is visible. Or use
  `tsx` for `start` in dev and a separate `start:prod`
  for the built output.
