R56 Web Audit Findings ŌĆö 2026-06-06
===================================

Executive summary
-----------------
Audited apps/web/{app,components,hooks,lib} for end-to-end integration gaps
that survived the R55 sweep. Found 22 high-quality issues:

  CRITICAL: 0
  HIGH    : 6
  MEDIUM  : 12
  LOW     : 4

Top survivors:
- Coin selection (`objects[0]` instead of largest-by-balance) in two
  pages the R55 sweep fixed at other call sites but missed.
- Per-route `loading.tsx` / `error.tsx` files still missing (deferred
  in R55; re-flagging now).
- A raw `dAppKit.signAndExecuteTransaction` call in the legacy trade
  page that the R55 sweep was supposed to migrate to `submitAndWait`.
- Two `JSON.parse(localStorage)` calls without try/catch.
- A `client!` non-null assertion in two streak components that can
  throw on click when dapp-kit is still initializing.

Findings

[HIGH] R56.1 | apps/web/app/markets/[id]/page.tsx:1060 | `redeemWinner` picks `objects[0]` instead of the largest winning-coin fragment
  Where: apps/web/app/markets/[id]/page.tsx:1060
  What: After the R52 audit raised `limit: 100` so a winner with many
        winning-coin fragments is enumerated correctly, this survivor
        still picks `const coin = objects[0];` ŌĆö the first coin in the
        indexer's response order, not the largest. A user with ten
        0.1-share winning-coin fragments and one 0.5-share fragment
        will see the redeem use a 0.1 fragment first, abort on-chain
        with `EInsufficientBalance` (the on-chain `redeem` redeems
        the whole input coin), pay gas, and need to retry until the
        largest fragment happens to land first.
  Why:  Silent failure of the redeem flow for any user with more
        than one winning-coin fragment (which the R52 limit bump
        made more likely, not less).
  How:  Mirror the `splitCollateral` pattern at line 522-524:
        const sorted = [...objects].sort((a, b) =>
          BigInt(b.balance) > BigInt(a.balance) ? 1 : -1,
        );
        const coin = sorted[0];

[HIGH] R56.2 | apps/web/app/vault/page.tsx:114 | VLP `refreshPlp` effect picks the first VLP coin, not the largest
  Where: apps/web/app/vault/page.tsx:114
  What: `setVlpCoinId(objects[0]?.objectId ?? "");` selects the
        first VLP coin in the indexer's response order, not the
        largest by balance. R53 raised the page-size limit to 100
        but did not add a largest-coin sort. The withdraw handler
        then passes this `vlpCoinId` to `buildVaultWithdrawTx` ŌåÆ
        `tx.object(vlpCoinId)`. A user with 5 VLP coins (typical
        after several deposits) withdraws from the smallest, which
        on the withdraw path triggers `EInsufficientBalance` for
        any withdraw > smallest-coin-balance.
  Why:  Silent failure of withdraw for any dust-fragmented user;
        the deposit path (line 155-158) was fixed but the
        VLP-side effect was missed.
  How:  Same fix as R56.1: sort `[...objects]` by `BigInt(b.balance)`
        and pick `sorted[0]?.objectId ?? ""`.

[HIGH] R56.3 | apps/web/app/legacy/predict/vault/page.tsx:126 | `supplyPLP` uses `coins.objects[0]!` as the merge target
  Where: apps/web/app/legacy/predict/vault/page.tsx:126
  What: `const primary = tx.object(coins.objects[0]!.objectId);`
        followed by `if (coins.objects.length > 1) tx.mergeCoins(
          primary, coins.objects.slice(1)...)`. The merge
        consolidates everything into whatever coin the indexer
        returned first, which can be a tiny dust coin. The
        subsequent `tx.splitCoins(primary, [tx.pure.u64(supplyAmount)])`
        then asks the chain to split more atoms than `primary`
        holds ŌåÆ on-chain `EInsufficientBalance` abort.
  Why:  The R55 sweep audited the modern vault but missed the
        legacy PLP vault. A user with several small DUSDC coins
        hits the same `EInsufficientBalance` they would have on
        the un-audited code path.
  How:  Pick the largest coin as `primary`:
        const sorted = [...coins.objects].sort((a, b) =>
          BigInt(b.balance) > BigInt(a.balance) ? 1 : -1,
        );
        const primary = tx.object(sorted[0]!.objectId);

[HIGH] R56.5 | apps/web/app/legacy/predict/trade/page.tsx:88 | `createManager` still uses raw `dAppKit.signAndExecuteTransaction` (R55 sweep survivor)
  Where: apps/web/app/legacy/predict/trade/page.tsx:88
  What: `const result = await dAppKit.signAndExecuteTransaction({
          transaction: tx });` followed by a manual
        `await client.waitForTransaction({ digest: result.Transaction.digest });`
        on line 105. The sibling `mintPosition` (line 251) and
        `redeemPosition` (line 310) were migrated to
        `submitAndWait(dAppKit, client, tx)` in R55 ŌĆö `createManager`
        was the survivor. The result is a manual
        `waitForTransaction` call that doesn't pass a timeout or
        `AbortSignal`, so a slow RPC could hang the modal
        indefinitely. The `mint` path (line 251) routes through
        `submitAndWait` and gets the 30s timeout + signal plumbing
        for free.
  Why:  Inconsistent error handling, missing timeout, and the
        R55 sweep's stated goal ("every modern flow is fixed")
        was not actually achieved here.
  How:  `const result = await submitAndWait(dAppKit, client, tx);`
        and read `result.digest` (with the same `$kind !== "Transaction"
        guard as the other handlers).

[HIGH] R56.6 | apps/web/app/dispute/[marketId]/page.tsx:134-177 | No `invalidateQueries` after a successful dispute

[MEDIUM] R56.8 | apps/web/components/StreakWelcomeBanner.tsx:51,72 | `JSON.parse(localStorage)` without try/catch
  Where: apps/web/components/StreakWelcomeBanner.tsx:51,72
  What: `const dismissedAddrs: string[] = raw ? (JSON.parse(raw)
        as string[]) : [];` (line 51) and the same in `dismiss()`
        (line 72). A user with corrupted localStorage
        (manually edited, or a future migration that wrote
        malformed JSON) would crash the effect on mount
        (no banner ever shows) or crash the "Later" click

  Why:  R55 audited the auth page's URL parsing (line 25-27)

[MEDIUM] R56.9 | apps/web/components/StreakProfile.tsx:70,205 & apps/web/components/StreakWelcomeBanner.tsx:91 | `client!` non-null assertion can throw when dapp-kit is still initializing
  Where: apps/web/components/StreakProfile.tsx:70,
         apps/web/components/StreakProfile.tsx:205,
         apps/web/components/StreakWelcomeBanner.tsx:91
  What: `await submitAndWait(dAppKit, client!, tx);` uses a
        non-null assertion on the result of `useCurrentClient()`.
        The buttons are rendered when `!streakId && account` ŌĆö
        `account` is the only gate. `client` can legitimately
        be `null` while dapp-kit is initializing (race on
        initial mount) or after a wallet disconnect mid-render.
        `client!` then throws `TypeError: Cannot read
        properties of null (reading 'waitForTransaction')`
        inside `submitAndWait`, the catch fires, the user
        sees "Streak creation failed: Cannot readŌĆ”", and
        the only fix is a page reload.
  Why:  Sibling call sites in `markets/[id]/page.tsx`,
        `vault/page.tsx`, and `parlay/page.tsx` all gate
        on `if (!account || !client || !...) return;` BEFORE
        calling `submitAndWait`. The two streak components
        missed this guard.
  How:  Wrap each click handler with
        `if (!client) { toast.error("Wallet not ready"); return; }`
        and remove the `!` from `client!`.

[LOW] R56.18 | apps/web/app/markets/[id]/page.tsx:626 | `clientOrderId = String(Date.now())` is millisecond-granular
  Where: apps/web/app/markets/[id]/page.tsx:626
  What: `const clientOrderId = String(Date.now());` is used
        as the on-chain client-order-id. Two orders
        submitted in the same millisecond would collide
        and the indexer's `chain_orders` de-duplication

        user's `clientOrderId`. The odds are low but the

        `crypto.randomUUID()`).
