R57 Web Audit — End-to-End Integration Gaps (Next.js / apps/web)
==================================================================
Scope: Next.js 14 App Router web app at apps/web/. Consumes @suipredict/sdk
and the agents REST API at NEXT_PUBLIC_AGENTS_URL. Reads .r56-web-findings.txt
for context on the R56 sweep that just landed.

Summary
-------
After 57 rounds the codebase is in very good shape. The R55/R56 sweeps
have driven `submitAndWait` to every call site, normalised coin
selection to largest-first, gated every poll/interval on tab
visibility, and wrapped the streak-banner / parlay-history fetches
in AbortController. This round finds ~17 surviving issues, mostly
in the markets/[id] trade page (which got the most R56 attention
and therefore had the most regression surface). No new CRITICALs,
2 HIGH, 7 MEDIUM, 8 LOW.

HIGH
----
R57.H1  markets/[id]/page.tsx:805-866  waitForOrderInBook: visibility-gate
       promise leaks its `visibilitychange` listener if the caller aborts
       while the tab is hidden.
       - Bug: the `await new Promise(...)` on lines 854-862 attaches
         `addEventListener("visibilitychange", onVis)` but never removes
         the listener if the outer loop's `signal?.aborted` check on
         line 820 fires (which it does — the signal check runs BEFORE
         the visibility-gate). The promise is GC-able but the listener
         holds a strong reference to its `onVis` closure, which in turn
         holds the surrounding `resolve`. Over a long session with
         several mints that all fire while the tab is backgrounded,
         listeners pile up.
       - Why: a hidden tab + a wallet in another window + 5 placeOrder
         cycles ends with 5 dangling listeners. Each will only ever
         fire once, but a subsequent visibility flip fires all 5 and
         each tries to resolve a Promise the loop has already abandoned
         — silent for now but will surface as a useEffect-recompute
         storm in the next refactor.
       - Fix: factor the visibility wait into a helper that takes the
         outer `signal` and resolves on `signal.aborted` as well as
         `visibilitychange → "visible"`. Clean up the listener in BOTH
         branches.

R57.H2  markets/[id]/page.tsx:908-919  createBalanceManager: silent
       null swallow on `extractCreatedObjectId`.
       - Bug: lines 913-919 only persist the manager id to state +
         localStorage if `managerId` is truthy. The success toast on
         line 920 fires regardless, so the user sees "BalanceManager
         created: <digest>..." even when the gRPC query for the
         new shared object id returned null (RPC timed out, object
         query indexer lag, etc). The "Deposit" / "Place Order" buttons
         then disable themselves on `!balanceManagerId` and the user
         has no clue why the success toast didn't unlock them.
       - Why: the previous block (lines 897-904) already handles the
         real failure case (`r.$kind !== "Transaction"`, no digest).
         This is the *post-success* silent failure — same UX impact
         as the failed case, half the error handling.
       - Fix: if `managerId` is null after a Transaction-kind result,
         toast a distinct "BalanceManager created but id not yet
         visible to this RPC node — refresh in a few seconds" message
         and schedule a retry-on-refresh (already implicit via
         setRefreshCounter but explicit retry would be better).

MEDIUM
------
R57.M1  markets/[id]/page.tsx:401-413  localStorage.setItem without
       try/catch.
       - Bug: the two effects on lines 401-406 and 408-413 call
         `window.localStorage.setItem` unconditionally. In Safari
         private mode, Brave strict mode, and any iframe with
         `allow-same-origin` removed, the call throws
         `QuotaExceededError` synchronously and the effect tears
         down the entire `useEffect` chain — `setBalanceManagerId`
         / `setTradeCapId` become dead writes for the rest of the
         session.
       - Why: every other "persist a wallet-scoped key" path
         (auth/page.tsx, providers-inner.tsx) wraps the write in
         try/catch. This is the only path that doesn't.
       - Fix: try { localStorage.setItem(...) } catch (e) {
         console.warn(...) }.

R57.M2  markets/[id]/page.tsx:461-467  setRefreshCounter cascade
       from deepBookMarket effect.
       - Bug: the effect mirrors `deepBookMarket` into a ref and,
         when truthy, fires `setRefreshCounter(c => c + 1)`. This
         effect re-runs whenever `deepBookMarket` changes, but
         `deepBookMarket` itself is a fresh useMemo result if
         `market` is undefined on first render then becomes defined
         on the second. The `setRefreshCounter` fires twice (once
         for the null→defined transition, once for the first
         `account` change), each of which re-runs the
         `refreshBookAndOrders` useEffect on `[account, marketId,
         refreshCounter]`, each of which fetches the order book
         and the open-orders list in parallel. Three redundant
         fetches on initial mount with a real market.
       - Why: the comment on line 464 says "Trigger an immediate
         re-read now that the pool is known" — that's the
         intentional first call. The second call is the regression.
       - Fix: gate with a `useRef(false)` "initialized" flag. Only
         call `setRefreshCounter` on the first true→true transition.

R57.M3  parlay/page.tsx:322-324  in-place `objects.sort()` mutates
       SDK response.
       - Bug: line 322 calls `objects.sort(...)` directly. The
         `objects` array comes from `client.core.listCoins(...)` on
         line 308 — depending on the SDK implementation, this may
         be the cached array returned to the next `listCoins` call
         in the same component (the R43 `Promise.allSettled`
         parallel list), and sorting in place will silently change
         the order the other consumer saw. In practice the Sui
         SDK returns a fresh array per call, so the bug is latent,
         but R55's fix to vault/page.tsx used the safer
         `[...objects].sort(...)` spread. This call site didn't get
         the spread.
       - Why: latent ordering bug, fixes for it are cheap.
       - Fix: `const sorted = [...objects].sort((a,b) => ...);
              const coin = sorted[0]!;`

R57.M4  components/DailyPredictionCard.tsx:175-177  same
       in-place sort pattern.
       - Bug: identical to R57.M3. The `objects` from
         `client.core.listCoins` is sorted in place. Comment on
         line 171 says "Pick the largest coin" — the comment is
         correct, the implementation just mutates a shared array.
       - Fix: same as R57.M3.

R57.M5  components/ParlayHistory.tsx:102-129  toggleDetail fetch
       without AbortController.
       - Bug: the list-endpoint fetch on lines 84-95 was wrapped
         in AbortController by R56.20. The single-row
         `toggleDetail` fetch on lines 102-129 was missed —
         clicking a row, then immediately clicking it again, then
         clicking a third row, fires three concurrent fetches
         against `/parlay/{id}` with no cancellation. The 2nd
         and 3rd responses race the 1st and the last one to
         resolve wins the state write. Mid-network-slow
         (mobile / Enoki), the user sees the wrong row's detail
         in the expanded pane.
       - Why: the `expanded` map keyed by `parlayId` is meant
         to isolate per-row state, but the last-resolved fetch
         still overwrites `expanded[parlayId]`. A second click
         on row A while row B's fetch is in flight writes
         `expanded[A] = rowA_fetch1` then `expanded[A] =
         rowA_fetch2`, but if the user clicked A → B → A and B
         resolves last, `expanded[A] = rowB_payload` briefly.
       - Fix: add a per-row AbortController (Map<parlayId, ctl>)
         aborted on each click. If `expanded[parlayId]` is already
         loaded, abort any in-flight before collapsing.

R57.M6  components/StreakProfile.tsx:214  `streakId!` non-null
       assertion in inline onClick handler.
       - Bug: the inline onClick on line 192 calls
         `buildMintBadgeTx({ streakId: streakId!, tier })`. The
         R56 `!client` guard is in place on line 200 but the
         `streakId!` non-null assertion is not guarded. If the
         streak id refetch (after a new tx invalidate) returns
         null mid-render (the R43 staleTime race), the assertion
         passes `undefined` to the SDK builder, which constructs
         a tx with `tx.object(undefined)` and the wallet surfaces
         an opaque "Invalid Object Id" error.
       - Why: the parent on line 50-121 guards `!streakId` and
         renders a different component if it's null. The button
         is only mounted when `streakId` is truthy, so the
         assertion is safe in the happy path. The race is
         during the gap between the parent deciding to mount
         the button and the React 18 commit — a click on the
         still-mounted button after the streak id refetched
         to null would fire with the stale closure.
       - Fix: `if (!streakId) return;` at the top of the handler,
         same as the surrounding `if (!client)` guard.

R57.M7  app/agents/page.tsx:312-329  `let id = setInterval(...)`
       reassigned inside the tick closure (R56.19 survivor).
       - Bug: the R56.19 fix was to switch from `setTimeout`-
       chained polling to a `setInterval` that re-arms itself
       with a new interval id when the backoff changes. The
       implementation pattern works on the happy path but has
       two residual issues:
         (a) The `let id` declaration on line 330 is used by
             the `clearInterval(id)` on line 324 BEFORE its
             initial assignment completes (the `let id` is
             hoisted to the top of the closure scope, the
             actual `setInterval` call on line 330 runs after
             `tick()` is wired). On the very first tick if
             `errorRef.current` is already true (the mount
             effect's `load()` failed before this code runs),
             `tick()` reads `next = 30_000`, sees
             `next !== backoffMs`, calls `clearInterval(id)`
             where `id` is `undefined` — a no-op — then
             assigns `id = setInterval(tick, 30_000)` for
             the FIRST time. The initial `setInterval(tick,
             8000)` on line 330 immediately fires the second
             `tick` 8s later, racing the new 30s interval.
         (b) The cleanup on line 333 always calls
             `clearInterval(id)`. If the user unmounts
             mid-tick, the in-flight `load()` continues and
             its setState will be a setState-after-unmount
             warning. The `cancelled` flag on line 314 only
             gates the next-tick, not the in-flight fetch.
       - Why: the original setTimeout-chained version was
       cleaner about cancellation. The setInterval migration
       traded clarity for fewer timers.
       - Fix: revert to setTimeout-chained: `const tick =
       () => cancelled ? undefined : setTimeout(() => {
         ... ; tick() }, backoffMs);` and add an
       `AbortController` to the in-flight `load()` so the
       unmount cleanup can `ctl.abort()`.

R57.L4  app/legacy/predict/trade/page.tsx:48-78  useEffects

R57.L5  app/agents/page.tsx (R56.19 survivor — see R57.M7)  also
       has a stale `let backoffMs = 8000` that lives in the
       closure but is captured by the dep array comment on
       line 337 saying `error` is the "only state-derived
       dependency". Technically accurate (it's a local var,
       not state) but the effect re-binds on `error` changes
       purely to re-evaluate `backoffMs`, and each rebind
       resets `backoffMs` to 8000 — so an error cycle of
       "5xx → 200 → 5xx → 200" cycles `backoffMs` through
       "8s → 30s → 8s → 30s" with each transition firing
       the `clearInterval` + `setInterval` dance. R57.M7's
       setTimeout-chained revert is the right fix.

R57.L6  app/auth/page.tsx:17  enokiFlow.handleAuthCallback()
       has no timeout.
       - Bug: the `useEnokiFlow().handleAuthCallback()` call
       on line 17 returns a Promise that resolves only when
       the zkLogin JWT round-trip completes. If the user's
       network drops after the wallet-redirect-comes-back
       step, the Promise hangs forever, the page sits on the
       "Completing sign in..." spinner, and the user has to
       manually reload. The `cancelled` flag only stops
       setState, not the in-flight fetch.
       - Fix: race the call against a 15s timeout, surface
       a "Sign in timed out — try again" toast on lose.

R57.L7  app/vault/page.tsx:84-110  polling interval not aborted
       on unmount (continues running after navigation).
       - Bug: the `useEffect` on lines 84-110 returns a
       cleanup that calls `clearInterval(t)`, but the
       `refresh()` promise itself is not aborted. If the
       unmount happens between `await getVaultSummaryClob()`
       and `setSummary(...)`, the setState fires on the
       unmounted component. Not a leak (the setState is
       a no-op on unmounted React 18 components) but the
       `refresh()` RPC completes and the result is
       discarded.
       - Why: low impact, but R56.15 added the visibility
       gate and 30s backoff to this same effect; an
       AbortController was the natural next step.
       - Fix: thread an AbortController into the
       `getVaultSummaryClob` call (if the SDK supports
       signal) or wrap refresh in a `cancelled` flag.

Per-route loading.tsx / error.tsx still missing
------------------------------------------------
R55 deferred this finding, R56 re-flagged it. R57 re-flags it
once more. The Next.js App Router convention is to put
`app/<route>/loading.tsx` + `app/<route>/error.tsx` next to
each `page.tsx` so navigations between routes show a skeleton
instead of the previous route's frozen state, and so a server
component error in /markets/[id] doesn't bubble all the way to
the root error boundary. As of R57:
  - /vault — missing
  - /markets — missing
  - /markets/[id] — missing (largest, most likely to throw)
  - /dispute/[marketId] — missing
  - /parlay — missing
  - /portfolio — missing
  - /agents — missing
  - /admin — missing
  - /settings — missing
  - /profile — missing
  - /auth — missing
  - /leaderboard — missing
  - /legacy/predict/* — missing

The root `app/loading.tsx` + `app/error.tsx` cover the cold
mount but not the in-app navigation case. Low priority, but
worth a sprint.

Not regressions
---------------
- `submitAndWait` migration is complete. No raw
  `dAppKit.signAndExecuteTransaction` survivors found in the
  22 page files audited.
- `client!` non-null assertions are all gated in production
  code paths. R56.9 added guards in StreakWelcomeBanner and
  StreakProfile; no third survivor found in R57.
- Coin selection sort is consistently applied via
  `BigInt(b.balance) > BigInt(a.balance)` across vault,
  parlay, and DailyPredictionCard. R57.M3 / R57.M4 are the
  in-place-mutation variant of the same pattern; the
  *sorting* is correct, the *mutation* is the issue.
- Visibility-gated polling: agents, vault, markets/[id] order
  confirm, leaderboard all gate on `document.visibilityState
  === "visible"`. Consistent.
- Streak cache invalidation: createStreak, claimBadge, and
  revokePolicy all use `queryClient.invalidateQueries({
  queryKey: ["userStreakId"] | ["streakInfo"], type:
  "active" })`. R52/R54/R56.7 fixes are in place.
- AbortController on long fetches: parlay/user history (R56.20)
  and markets/[id] order confirm (R36) have it. R57.M5 is
  the toggleDetail survivor.
- $kind guards on `submitAndWait` results are consistent
  across all call sites (R38 + R55 cycle).
