Round 56 Agents Audit - Executive Summary
=========================================
Total findings: 28
- CRITICAL: 3
- HIGH: 9
- MEDIUM: 12
- LOW: 4

The R55 sweep's `safeInt`/`safeFloat`/`safeBigInt` helpers were not retrofitted to all module-level `Number(process.env.X)` sites. The R47 audit's claim that the demo order counter is SQLite-backed is not implemented in code. Several places still log generic LLM failures that hide the real cause. One new integration gap: POST /prize/claims and /prize/signature derive the prize amount from different sources, silently desyncing the off-chain mirror from the on-chain payout.

[CRITICAL] §R56.01 | apps/agents/src/agents/market-maker.ts:74-85 | demo order counter is in-memory Map, not SQLite-backed as R47 claimed
  Where: /Users/choguun/Documents/workspaces/hackathon/SuiPredict-AI/apps/agents/src/agents/market-maker.ts:74
  What: `const demoOrderCounters = new Map<string, number>();` is a module-level in-memory Map. The R47 audit comment at line 67-70 explicitly claims "The counter lives in a SQLite-backed module map so it survives restarts" but the implementation is a plain in-memory Map.
  Why: A Railway redeploy or SIGTERM/RESTART clears the Map. On the next tick, `nextDemoOrderId(marketId)` re-initializes from `Date.now()`, which on a fast redeploy may be the same or a smaller value than a previously-written `order_id`. Two `upsertOrder` calls in the same (market_id, order_id) primary key collide silently — the second `ON CONFLICT` updates `filled=0` and `quantity=`, which can re-flip a previously-cancelled demo order back to "open", and the order-book REST surface shows phantom orders until the next tick.
  How: Persist the counter to the `markets.db` (e.g. a `demo_order_counters` table with `(market_id, next_id)`, seeded from `MAX(order_id)+1` on first use) and read/update inside `nextDemoOrderId`. Alternatively, drive the id from a SQLite `AUTOINCREMENT` PK by inserting a parent row and reading back `last_insert_rowid`.

[CRITICAL] §R56.02 | apps/agents/src/markets/store.ts:788-789 | getVaultSummaryFromEnv uses raw Number() on env, R55 missed
  Where: /Users/choguun/Documents/workspaces/hackathon/SuiPredict-AI/apps/agents/src/markets/store.ts:786
  What: `getVaultSummaryFromEnv()` does `Number(process.env.VAULT_TOTAL_BALANCE ?? 0)` and `Number(process.env.VAULT_ALLOCATED ?? 0)`. The R55 sweep added `safeFloat` to lib.ts specifically for this pattern but `getVaultSummaryFromEnv` was missed.
  Why: A misconfigured env value (`VAULT_TOTAL_BALANCE=10_000_000` pastes a `10_USDC` suffix → `Number("10_USDC")=NaN`; or a `1e20` value that exceeds MAX_SAFE_INTEGER) silently corrupts the risk monitor's utilization calc. With `total=NaN`, the `total_balance > 0 ? allocated / total_balance : 0` branch falls to 0% utilization, so the risk monitor never trips the pause threshold even when the vault is critically over-utilized. The opposite (`total=0` from missing env + `allocated=10_000_000`) reports 0% utilization when the vault is actually over-budget.
  How: Route through `safeFloat(process.env.VAULT_TOTAL_BALANCE, 0)` and `safeFloat(process.env.VAULT_ALLOCATED, 0)`, and log a warning when the parsed value is non-finite. Add a `getVaultSummaryFromEnv` entry to the boot-config validator in index.ts so a fresh deploy fails loud.

[CRITICAL] §R56.03 | apps/agents/src/gamification/routes.ts:903-914 | POST /prize/claims derives amount from env, /prize/signature derives from on-chain → mirror desync
  Where: /Users/choguun/Documents/workspaces/hackathon/SuiPredict-AI/apps/agents/src/gamification/routes.ts:903
  What: The signature route (`/prize/signature`) computes `amount` via `resolvePrizeAmount()` which reads the on-chain `PrizePool.weekly_prize` first and falls back to env. The POST claims route computes `amount` solely from `safeBigInt(process.env.PRIZE_WEEKLY_AMOUNT, 0n)`. Same `(user, week, rank, poolId)` → two different `amount` values.
  Why: When the on-chain `weekly_prize` diverges from the env (the operator funds the pool directly out-of-band, or `PRIZE_WEEKLY_AMOUNT` is stale after a manual `fund_pool` tx), the user's signed claim payload carries the on-chain amount (what the contract will actually pay), but the off-chain `prize_claims.amount` column records the env-derived amount. The mirror row disagrees with the on-chain payout forever. An auditor reconciling the two surfaces sees a phantom "claimed $X, paid $Y" mismatch with no way to resolve.
  How: Extract a `resolvePrizeAmount(poolId, week, rank)` helper that both routes call, identical to the one already in `/prize/signature`. The POST handler should call the same helper and write the same `amount` to the mirror.
